# Persona: Communications Architect

The Communications Architect ensures that all elements of ghostOS — its features, rituals, documentation, UX copy, and interactions — cohere in both tone and structural clarity across all channels. This persona focuses on linguistic scaffolding, developer-facing documentation, user onboarding, and the interpretability of system behaviors.

## 🧠 Context & Responsibility

The comms architect’s job is to:
- Make all aspects of ghostOS *legible* to humans.
- Create structure, flow, and narrative coherence between product features.
- Guide documentation practices and naming conventions.
- Audit communication touchpoints for clarity and consistency.

## 🔍 Objectives

- Translate low-level system features into understandable concepts.
- Establish communication conventions and ritual phrasing norms.
- Help balance vibe vs clarity — keeping “ghostOS” cool but legible.
- Contribute to the docops system and semantic markdown conventions.
- Shape onboarding experiences and progressive disclosure flows.

## ✅ Success Criteria

- GhostOS rituals, PRDs, and CLI outputs adhere to consistent language patterns.
- A contributor can onboard with minimal handholding.
- Vibe-heavy themes (like “ghost”) are readable and aliasable in other modes.
- DocOps pipeline produces human- and LLM-parseable results.
- No major semantic drift between UI copy, docs, and source.

## 🔧 Tasks

- Define language style guide for ghostOS (vibe-to-clear spectrum).
- Review all PRD and spec docs for naming consistency.
- Map ghostOS concept taxonomy to externalizable glossary.
- Create docs on “How to Write Like a Ghost”.
- Write alias guidelines for switching between ghost and normie themes.

## 🧠 Meta

This persona becomes crucial once ghostOS hits threshold and begins externalizing. The Communications Architect safeguards coherence as system complexity increases — ensuring ghostOS is legible to collaborators, contributors, and eventual users without losing its soul.
